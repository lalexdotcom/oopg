    **A. Create TransactionClient class (REFAC-01, per D-06/D-07/D-08):**

    Add a `TransactionClient` class at the bottom of `src/database.ts` (before the entity helper classes, after the `Database` class body). Do NOT export it from `index.ts` (per D-04 — it is internal).

    ```typescript
    class TransactionClient extends Database {
      #done = false;
      #client: PoolClient;
      #parentPool: Pool;

      constructor(client: PoolClient, parentDb: Database) {
        // Call super with the parent's config. This creates a Pool that will never be used.
        // pg.Pool is lazy (no connections until connect() is called), so the unused Pool
        // is garbage collected harmlessly.
        super(parentDb.config);
        this.#client = client;
        this.#parentPool = parentDb.pool;
        // Immediately end the unused pool created by super() to prevent resource leak
        this.pool.end();
      }

      // Override pool getter to route connect() to the held transaction client (per D-06)
      // When #done is true (after commit/rollback), fall back to parent pool
      get pool(): Pool {
        if (this.#done) return this.#parentPool;
        // Return a Pool-shaped object whose connect() resolves the unreleasable client
        const client = this.#client;
        return {
          connect: () => Promise.resolve(client),
          // Delegate pool status properties to parent pool for tx.status compatibility (Pitfall 3)
          get idleCount() { return 0; },
          get waitingCount() { return 0; },
          get totalCount() { return 1; },
        } as unknown as Pool;
      }

      // Make pool a getter by ensuring it shadows the parent's pool field
      // NOTE: The executor must ensure pool is declared with `get pool()` override,
      // which requires the parent Database.pool to be accessed as a property.
      // Since Database declares `pool: Pool` as a public field, the getter override
      // works in TypeScript (property -> getter override in subclass).

      // Override transaction() to throw (per D-06) — prevents nested transactions
      override async transaction(): Promise<never> {
        throw new Error('Cannot initiate nested transaction');
      }

      // Mark transaction as done (called by commit/rollback)
      markDone(): void {
        this.#done = true;
      }

      get isDone(): boolean {
        return this.#done;
      }
    }
    ```

Alternatively, override `release` in the pool-shaped object's connect() return value. The executor should choose the approach that keeps the code simplest.

It IS exported from `database.ts` with `export class` so that `instanceof` checks work from consumer code that imports `TransactionClient` directly from `src/database` (the ROADMAP success criterion SC-2 requires `instanceof TransactionClient` to return true).     **B. Rewrite Database.transaction() to use TransactionClient (REFAC-01, per D-06/D-07/D-08):**

    Replace the entire `transaction()` method body (lines 240-330) with:

    ```typescript
    async transaction<T>(
      callback: (
        transaction: this,
        functions: { commit: () => Promise<void>; rollback: () => Promise<void> },
      ) => Promise<T> | T,
      options?: TransactionOptions,
    ) {
      const { autoCommit, ...opOptions } = options ?? {};
      const client = await this.pool.connect();

      // Create unreleasable client wrapper (release is handled in finally block)
      const originalRelease = client.release.bind(client);
      client.release = () => {};  // no-op during transaction

      const txClient = new TransactionClient(client, this);

      const commit = async () => {
        if (txClient.isDone) return;  // idempotent per BUG-03
        await execute(client, 'COMMIT', opOptions);
        txClient.markDone();
      };
      const rollback = async () => {
        if (txClient.isDone) return;  // idempotent per BUG-03
        await execute(client, 'ROLLBACK', opOptions);
        txClient.markDone();
      };

      try {
        await execute(client, 'BEGIN', opOptions);
        const returned = await callback(txClient as unknown as this, { commit, rollback });
        if (!txClient.isDone) {
          if (autoCommit) {
            await commit();
          } else {
            await rollback();
          }
        }
        return returned;
      } catch (e) {
        if (!txClient.isDone) await rollback();
        throw e;
      } finally {
        client.release = originalRelease;  // restore original release
        client.release();
      }
    }
    ```

    Key changes from the current implementation:
    - `this.#done` on Database is no longer used for transaction state (D-08) — `txClient.isDone` / `txClient.markDone()` replace it
    - All 4 Proxy layers are removed — no `new Proxy(...)` in transaction()
    - Entity routing works automatically: when `callback` receives `txClient`, any `txClient.table(...)` call passes `txClient` (the TransactionClient) as `this` to the `TableImpl` constructor, so `entity.database = txClient`. Since `txClient.pool.connect()` returns the held PoolClient, all entity queries route through the transaction.
    - For schema entities built on the parent Database (e.g., `db.schema(...)` objects accessed as `tx.schemaProperty`), the current Proxy intercepts these. With TransactionClient, schema properties defined on the parent Database instance are inherited. The schema's entity helpers have `database = db` (the parent). This means `tx.schemaProperty.items.find()` will NOT automatically route through the transaction client without additional logic. The executor must handle this — two options:
      a) Override property access with a single Proxy on `TransactionClient` that intercepts schema/entity property reads and rewires `entity.database` to `this` (simpler than 4 nested Proxies — just one)
      b) Copy schema/entity properties during TransactionClient construction, rebuilding them with `this` as the database reference

    The executor should check the Plan 01 integration test results: if the `tx schema() entities route queries through transaction client` test passed on the Proxy-based code, then the replacement must also pass it. If it failed (documented as known gap), the executor has discretion on whether to fix or defer.

    **D. Clean up connect() Proxy (REFAC-02, per D-10):**

    Replace the Proxy in `connect()` (lines 636-641) with a direct approach:

    ```typescript
    async connect<T>(
      callback: (client: ClientBase, release: () => void) => T,
      autoRelease = true,
    ) {
      const client = await this.pool.connect();
      // Save and override release to prevent callback from releasing the client
      const originalRelease = client.release.bind(client);
      client.release = () => {};  // no-op — release is managed below

      let released = false;
      const release = () => {
        if (!released) {
          released = true;
          client.release = originalRelease;  // restore before calling
          client.release();
        }
      };
      try {
        const result = await callback(client as ClientBase, release);
        if (autoRelease && !released) {
          release();
        }
        return result;
      } catch (e) {
        if (!released) release();
        throw e;
      }
    }
    ```

    This eliminates the Proxy entirely. The client's `release` method is temporarily overridden to a no-op, then restored when the actual release happens. The callback receives the real client object (not a Proxy), so all PoolClient methods (query, copyFrom, copyTo, on, etc.) work directly without delegation concerns (Pitfall 5).
